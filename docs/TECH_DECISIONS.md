# Tech Decisions

> Last updated: 2026-04-28 (TICKET-001)
> This file captures stack-level decisions for Moshimo Box. Update when you change a load-bearing default.

## 1. Stack

| Concern | Choice | Why |
|---|---|---|
| Framework | Next.js 15 (App Router) + TypeScript + React 19 | Single process serves the React UI and server-side route handlers. The OpenAI Agents SDK (#018, #021) runs natively in Node route handlers. Server components let dashboard widgets fetch on the server cleanly (#011, #014). |
| Bundler | Webpack (Next default), `--no-turbopack` | Turbopack still has rough edges with cross-`src/` resolution and CSS imports. Revisit at TICKET-024 (perf pass). |
| Package manager | pnpm | Fast, disk-efficient, well-supported by Next.js docs. |
| Test runner | Vitest + React Testing Library + jsdom | TypeScript-first, fast, no Jest config friction. |
| Linter | ESLint flat config (`eslint-config-next`) | Ships with `create-next-app`. |
| Formatter | Prettier | Standard. Configured in `.prettierrc.json`. |
| Styling | Plain CSS + CSS Modules driven by design tokens — **no Tailwind** | The design system is already a tokens-first CSS system. Tailwind would add a parallel abstraction and risk drift. |
| Node version | 22 (see `.nvmrc`) | Current LTS as of 2026-04. |
| App location | Project root | The repo *is* the moshimo app. No monorepo motivation yet. |

Alternatives that were rejected:
- **Vite + Hono/Express** — two processes, CORS dance for nothing.
- **Remix** — smaller Agents SDK community.
- **Astro** — content-first, weaker for live dashboards.

## 2. Design tokens — the import contract

The canonical token file is [`docs/design/design-system/colors_and_type.css`](design/design-system/colors_and_type.css). The design-system directory is the source of truth for visual decisions across mocks, previews, and production.

Production code imports tokens from a **mirrored copy** at `src/styles/tokens.css`, not from the canonical file directly. This avoids cross-`src/` resolution issues with future bundler changes (notably Turbopack), and keeps the production build self-contained.

The mirror is regenerated by `pnpm sync:tokens`, which copies the canonical file verbatim and prepends a "do not hand-edit" header (see [`scripts/sync-tokens.mjs`](../scripts/sync-tokens.mjs)). Drift is the only failure mode; surfacing it via the script is the mitigation.

**Rule: all component CSS imports tokens via `src/styles/tokens.css`** (relative or via the `@/` alias once an alias-aware CSS pipeline is in place). No other source. If you change a token, edit the canonical file in `docs/design/design-system/`, then run `pnpm sync:tokens` and commit both.

Rollback path if the mirror approach becomes painful: replace `src/styles/tokens.css` with a one-line `@import url("../../docs/design/design-system/colors_and_type.css");` and delete `scripts/sync-tokens.mjs`. Both options are explicitly endorsed by the design system rules in `AGENTS.md`.

## 3. Why the CDN UI kit is reference-only

`docs/design/design-system/ui_kits/app/` is a hi-fi click-through built with React + Babel via CDN. It has no build pipeline, no TypeScript, no test infrastructure, and no module system.

It exists to **demonstrate** the dashboard product visually — the layout, motion, content tone, widget composition. Production reuses the same tokens and the same visual rules, but **rebuilds** every component as a real React + TypeScript module under `src/`. Nothing under `ui_kits/app/` should be imported, bundled, or shipped.

When implementing a widget that has a counterpart in `ui_kits/app/`, treat the click-through as a visual spec, not a code source: read it, re-derive the structure, and reimplement against the tokens.

## 4. Out of scope for MVP

These are explicitly not in the product or the codebase. Do not add libraries that imply them.

- User authentication or sessions
- Subscriptions, billing, payments
- Broker / trading integrations
- Multi-dashboard management or saved layouts beyond the in-memory session
- Mobile-specific responsive design beyond what falls out of the desktop layout
- Report export (PDF, CSV, image)
- Portfolio tracking, position management, or any user holdings
- Social, sharing, or collaboration features

## 5. Out of scope for TICKET-001 specifically

The scaffold installs only the minimum to satisfy `dev`, `build`, `test`, `lint`. It deliberately does **not** install:

- `lucide-react` (icons land with the chrome ticket #002)
- charting libraries — Recharts, visx, etc. (land with #013)
- `@openai/agents` and friends (land with #018)
- `zod` or other schema/validation libs (land with #003)
- state managers, data-fetching libs, i18n libs

These come with the tickets that need them. The "OpenAI Agents SDK integration path" is the route-handler architecture, not the dependency.
